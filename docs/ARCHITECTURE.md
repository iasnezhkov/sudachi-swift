# Architecture & custom surface

This document explains **what this project is, how it's put together, and —
importantly — what we add on top of `sudachi.rs` and why.** It is meant to be
read once by anyone evaluating, using, or maintaining the binding.

## What it is

A thin Swift binding to [`sudachi.rs`](https://github.com/WorksApplications/sudachi.rs)
(the Rust implementation of the Sudachi Japanese morphological analyzer),
generated with [UniFFI](https://github.com/mozilla/uniffi-rs). We do **not** fork
sudachi.rs — it is a pinned git submodule (`third_party/sudachi.rs`) and does the
actual analysis. Our code is a small wrapper crate plus generated Swift.

```
Swift app
   │  import Sudachi
   ▼
swift/Sudachi/…/Sudachi.swift   ← UniFFI-generated Swift (do not edit)
   │  C FFI (Sudachi.xcframework, a static lib)
   ▼
crates/sudachi-swift-uniffi/src/lib.rs   ← OUR wrapper (one small file)
   │  Rust API
   ▼
third_party/sudachi.rs (submodule)      ← the real tokenizer + dictionary loader
```

Three source-of-truth files define the surface:
- `crates/sudachi-swift-uniffi/src/lib.rs` — the wrapper implementation.
- `crates/sudachi-swift-uniffi/src/sudachi_swift.udl` — the UniFFI interface.
- `Sudachi.swift` — generated from the two above; regenerated by
  `scripts/build-ios.sh`.

## What sudachi.rs provides vs. what we add

**sudachi.rs does the real work:** morphological analysis, `mmap` dictionary
loading, the A/B/C split-mode logic, and all morpheme attributes (surface,
reading, dictionary/normalized form, part-of-speech, synonym group ids, OOV
detection, word ids, offsets).

**Our custom surface (everything Swift sees) is deliberately small and
additive:**

| Custom element | What it is | Why it exists |
|---|---|---|
| **Baked-in `default_config.json`** | `sudachi.json` embedded into the binary via `include_bytes!` | iOS can't load a config file or a plugin `.dylib`; we ship a config that uses only bundled plugin implementations, so the tokenizer "just works" on-device. |
| **`SudachiError` taxonomy** | 4 cases (`DictionaryNotFound` / `DictionaryInvalid` / `ConfigInvalid` / `Tokenization`), `LocalizedError` in Swift | sudachi.rs surfaces one opaque error; we classify it into actionable cases by matching on the error **variants** (`Io`/`ConfigError`/`InvalidHeader`/…), unwrapping `ErrWithContext`. Paths are also validated up front so a missing file is a precise `DictionaryNotFound`. |
| **`Morpheme` value type** | A flattened DTO copying 10 fields out of sudachi's borrowed `Morpheme` | Borrowed morphemes can't cross the FFI boundary; we own a plain struct. |
| **Codepoint offsets** | `begin`/`end` use `begin_c()`/`end_c()`, not byte offsets | Swift `String` is scalar/grapheme-indexed; byte offsets misalign on every kanji. This is the single most important correctness choice. |
| **`SplitMode` (A/B/C)** | Our FFI enum mapped to sudachi's `Mode` | Stable, Swift-native enum. |
| **`tokenize_with_mode`** | Re-tokenize a substring at a different granularity | UI use case: expand a Mode C compound into its Mode A parts on tap, without rerunning the whole pipeline. |
| **Thread-safety wrapper** | `Mutex<StatefulTokenizer>` (parking_lot) + `Arc<JapaneseDictionary>` | sudachi's `StatefulTokenizer` is single-threaded and mutable; the Mutex makes the Swift-facing tokenizer safe to share. Mode is saved/restored around each call so state never leaks between calls. |
| **Lean API — `MorphemeLite` + `tokenize_lite*`** | Compact morpheme (5 fields; `part_of_speech` pre-joined to a comma string; integer `pos_id`) | Halves per-morpheme allocations and FFI marshalling on hot display paths — we don't marshal 6 POS strings per token just to join them in Swift. |
| **Batch API — `tokenize_many*`** | Tokenize many strings in one FFI call under a single lock | Amortizes FFI crossing + lock cost across a whole document/catalog. |
| **`katakanaToHiragana`** | Free helper mapping U+30A1..U+30F6 down by 0x60 | Furigana display convenience. |
| **`Morpheme.range(in:)`** | Swift-only extension mapping codepoint offsets to `Range<String.Index>` via `unicodeScalars` | Saves every consumer from re-deriving (and getting wrong) scalar-offset → `String.Index` math. Lives in `Sudachi+Extensions.swift`, not the generated file. |
| **`SudachiDictionary(systemDictionary:userDictionaries:)`** | Swift-only convenience init taking file `URL`s | Matches the `fetch-dictionary.sh` layout (resource dir = the dictionary's parent), so app code goes from `Bundle.url` to a working dictionary in one line. Lives in `Sudachi+Extensions.swift`. |

Everything above is *additive* — the full `Morpheme` / `tokenize` path is
unchanged by the lean/batch additions.

## Concurrency

`SudachiTokenizer` wraps a mutable `StatefulTokenizer` in a `Mutex`, so a single
tokenizer instance serializes its calls. The dictionary
(`Arc<JapaneseDictionary>`) is immutable and `mmap`-backed, so the pattern for
parallelism is **one `SudachiDictionary`, many `SudachiTokenizer`s** — create one
tokenizer per worker/`Task` and run them concurrently over the shared dictionary.
The batch API (`tokenize_many`) intentionally holds one tokenizer's lock for the
whole batch (one FFI crossing); to parallelize a large corpus, shard it across
several tokenizers rather than relying on a single one.

## Build & distribution (summary)

- `scripts/build-ios.sh` cross-compiles the wrapper for three Apple targets —
  all arm64: iOS device, iOS simulator, macOS (x86_64/Intel slices are
  intentionally dropped; they'd roughly double the artifact, and Intel Macs can
  build from source) — runs `uniffi-bindgen`, strips the slices, and assembles
  `build/Sudachi.xcframework`.
- The package is consumed remotely as a **binary `.xcframework` target**
  (URL + checksum in the root `Package.swift`), the standard way Rust-backed
  Swift packages ship (cf. Mozilla application-services, Ferrostar).
- The dictionary is **not** part of the package; `scripts/fetch-dictionary.sh`
  downloads a SudachiDict edition, and the consuming app bundles it. See
  `NOTICE` for the attribution obligation that comes with redistributing a
  `.dic`.

## Maintenance implications (how the custom surface affects us)

The wrapper is small, but it couples tightly to sudachi.rs internals:
`ConfigBuilder`, `JapaneseDictionary::from_cfg`, `StatefulTokenizer`,
`MorphemeList`, `Mode`, and the `Morpheme` accessors (`begin_c`,
`part_of_speech_id`, `synonym_group_ids`, …).

Consequences to keep in mind:

- **sudachi.rs 0.7 is the migration that will touch this wrapper.** It is a
  binary-format ("v1") + loader rewrite and also renames the dictionary factory
  (`Dictionary.create()` → `Dictionary.tokenizer()`) and reworks config
  embedding. The exact call sites above are what a 0.7 bump has to update, plus
  a `.dic` rebuild. Until then we pin an older release on purpose.
- **The error taxonomy matches on sudachi's error *variants*** (not message
  text), so it is robust to locale and message wording. It still depends on the
  *set* of variants; if a 0.7 rework renames/removes variants, the `match` in
  `SudachiError::classify` must be revisited (unknown variants degrade to
  `Tokenization`, never crash — the enum is `#[non_exhaustive]`).
- **`Morpheme` and `MorphemeLite` duplicate fields** and must be kept in sync
  when the morpheme shape changes.
- **Generated code is committed** (`Sudachi.swift`) — never hand-edit it;
  change `lib.rs` + the `.udl` and regenerate.
- **`SudachiError` case names are UpperCamelCase** (`.DictionaryNotFound`) in
  Swift — UniFFI does not case-convert `[Error]` enum variants (unlike plain
  enums such as `SplitMode`). We surface them as-is rather than maintain a
  parallel wrapper enum that would break error identity across the FFI.
